# 工作文档搭子

工作文档搭子是一个住在飞书、钉钉、企业微信里的项目文档助手。

它不是让你再学一个知识库软件，也不是让团队多维护一个系统。它只做一件事：把散落在群聊、会议纪要、文档、PDF、网页链接里的信息，整理成一个会持续更新、能追溯来源、能回答问题的项目记忆。

> 发资料给它，它记住。  
> 问它问题，它回答。  
> 信息变了，它自己维护。  
> 有冲突，它来问你。

## 普通白领怎么用

你在项目群里直接 @ 工作文档搭子：

> @工作文档搭子 把这份会议纪要整理进 A 客户项目

它会回复：

> 已整理到「A 客户项目」。  
> 新增：客户需求、风险点、承诺事项、时间线。  
> 发现 1 个冲突：预算在两份资料中不一致，需要你确认。

第二天开会前，你问：

> @工作文档搭子 明天和 A 客户开会前我该注意什么？

它会回答：

> 你需要重点关注三件事：预算变化、法务顾虑、上线时间。  
> 上次你承诺补一版实施排期，目前资料中没有看到最终版本。  
> 建议会议开头先确认预算口径。  
> 来源：6 月 18 日会议纪要、报价草稿 v3、销售群聊天摘录。

周五写周报时，你问：

> @工作文档搭子 生成本周 A、B、C 三个项目的周报

它会从每个项目的记忆里整理进展、风险、下周计划，让你不用翻聊天记录、会议纪要和文档版本。

## 它适合谁

- 销售：记住客户需求、预算、竞争对手、承诺事项。
- 客户成功：跟踪客户问题、续约风险、历史沟通。
- 项目经理：整理进度、风险、决策、待办。
- 产品经理：沉淀访谈、需求、竞品分析、会议结论。
- 咨询顾问和研究员：持续整理资料，随时生成汇报。

第一阶段先服务一个核心场景：**项目型白领的个人/小团队工作记忆库**。

## 为什么优先接入飞书、钉钉、企业微信

因为白领的资料本来就在那里。

用户不应该为了“整理知识”打开一个新系统。更自然的方式是：

- 在飞书群里发会议纪要，它自动整理。
- 在企业微信群里问客户状态，它直接回答。
- 在钉钉项目群里上传文件，它维护项目记忆。
- 重要内容仍然放在企业已有的文档、云盘、知识库里。

所以这个项目的正式形态不是“本地 App 优先”，而是：

```text
飞书 / 钉钉 / 企业微信
        ↓
工作文档搭子平台接入层
        ↓
项目记忆引擎
        ↓
平台文档/云盘 + 索引 + 自动维护
```

本地页面只是方便开发和演示，不是主要使用方式。

## 数据放在哪里

第一阶段支持两种思路：

### 1. 平台优先存储

正式接入飞书时，推荐把用户能看到、能协作的内容放在飞书文档、飞书知识库或飞书云空间里。

工作文档搭子只保存：

- 哪个项目有哪些资料
- 每份资料的来源和哈希
- wiki 页面索引
- 待确认冲突
- 后台任务状态
- 必要的缓存

这样普通用户仍然在熟悉的飞书环境里查看和协作文档。

### 2. 本地验证存储

开发和演示时，也可以存在本地目录：

```text
data/
  local-demo/
    a客户项目/
      raw/
      extracted/
      wiki/
  state.sqlite
```

这只是为了快速验证产品闭环。

## 已完成的 MVP

当前版本已经包含：

- 本地验证页面：上传内容、问问题。
- 项目记忆引擎：把资料整理成项目总览、需求、风险、承诺、决策、人物、资料清单、待确认问题。
- 自动维护：新资料进来后自动重建项目记忆。
- 冲突提示：例如发现多个预算口径时，会提醒用户确认。
- 飞书 webhook adapter：可以接收飞书消息事件，把消息转换成“上传资料”或“问问题”。
- 存储抽象：后续可以把 wiki 页面写入飞书文档，而不是只存在本地。

## 飞书接入方式

第一版飞书接入的目标是让团队管理员能快速完成：

1. 在飞书开放平台创建企业自建应用。
2. 开启机器人能力。
3. 配置事件订阅，把消息事件推送到：

```text
https://你的服务域名/webhook/feishu
```

4. 配置环境变量：

```bash
FEISHU_APP_ID=你的应用 App ID
FEISHU_APP_SECRET=你的应用 App Secret
FEISHU_VERIFICATION_TOKEN=事件订阅 Verification Token
```

5. 把机器人加入项目群。

之后用户在群里 @ 工作文档搭子即可使用。

飞书、钉钉、企业微信本身已经提供消息、文件、卡片、文档和云盘相关能力。这个项目不会重复做聊天窗口、文件系统或审批 UI，而是尽量复用平台能力。

参考平台文档入口：

- 飞书开放平台：https://open.feishu.cn/document/home/index
- 钉钉开放平台：https://open.dingtalk.com/document/
- 企业微信开发文档：https://developer.work.weixin.qq.com/document

## 后续路线

### 第一阶段

## 简略技术说明

当前代码由四层组成：

```text
platform adapters   飞书、钉钉、企业微信、本地验证页
core engine         上传、整理、问答、自动维护
storage providers   本地存储、飞书文档/知识库预留
state.sqlite        索引、事件、冲突、会话记录
```

核心代码在：

```text
work_memory/engine.py
work_memory/compiler.py
work_memory/storage.py
work_memory/adapters/feishu.py
work_memory/server.py
```

后续接钉钉或企业微信时，不需要重写核心引擎，只需要新增 adapter，把平台消息转换成两种动作：

- 上传内容
- 问问题
